# xpfind · 快速上手

在混淆过的 App 里**按签名/结构定位被混淆的 Java 方法**，返回可反射调用的 `Method`。
**本库只定位，不做 Hook** —— 你拿到 `Hit.method`（`java.lang.reflect.Method`）后，用 Xposed/自己的框架去 Hook。

---

## 1. 最小例子

**基本流程：定位 → 拿 `Hit.method` → 自己 Hook。**

```java
// 在你的 Xposed 模块里，拿到 ctx 和目标 dex 的 ClassLoader 后：
List<Hit> hits = OL.with(ctx, cl)
        .pkg("com.target.app")                 // 必填：限定扫描的类名前缀（安全边界）
        .fast()                                // 已知精确签名用 FAST
        .find(Ri.m()
                .ret(String.class)
                .args(Context.class, String.class)
                .static_());

// 定位到了 —— Hook 由你自己做：
for (Hit h : hits) {
    XposedBridge.hookMethod(h.method, new XC_MethodHook() {
        @Override protected void afterHookedMethod(MethodHookParam p) {
            XposedBridge.log("hit " + h.sig() + " ret=" + p.getResult());
        }
    });
}
```

### 更多规则示例（都是 `OL.with(ctx, cl).pkg("…").<mode>().find(rule)` 里的那条 rule）

```java
// ① 无返回值（void）的实例方法：public void onEvent(String, Bundle)
Ri.m().ret(void.class).args(String.class, Bundle.class).public_()

// ② 返回值是父类型、不写死混淆名（推荐）：public ? xxx(String, String, Map)，返回某 Enum
Ri.m().retSuper(Enum.class).args(String.class, String.class, Map.class).public_()   // .fast()

// ③ 只知道参数个数，不知道类型：3 个参数的 public 方法
Ri.m().argc(3).public_()
Ri.m().min(2).max(4)                         // 或者 2~4 个参数

// ④ 静态工厂：static ? create(Context)，返回值是某 View 子类
Ri.m().retSuper(View.class).args(Context.class).static_()

// ⑤ 基本类型 / 数组参数：boolean check(int, byte[])
Ri.m().ret(boolean.class).args(int.class, byte[].class)

// ⑥ 按类结构找：类实现了某接口、且有个 public 无参方法
Ri.m().hasItf("Callback").argc(0).public_()

// ⑦ 类名/方法名含某子串（名字稳定时才用；混淆名会变，慎用）
Ri.m().hasCls("Manager").hasName("init").argc(0)

// ⑧ 返回值精确 + 只要第一个命中
Ri.m().ret("com.target.app.User").argc(0).first()

// ⑨ 自定义过滤（反射阶段，对候选再收窄）
Ri.m().args(String.class).filter(new Filter() {
    public String id() { return "has-2-fields"; }          // 稳定 id，进缓存 hash
    public boolean ok(Class<?> c, Method m) { return c.getDeclaredFields().length == 2; }
})
```

> `.fast()` = 类型字符串精确匹配（最快）；`.normal()` = 弱匹配（父类/子类/接口都算，返回父类型且 FAST 匹配不到时用）。`.retSuper()` 两种模式都在 dex 层生效。选型看 §2.5。

## 2. 三个核心概念

1. **只定位不 Hook**：`find()/run()` 返回 `List<Hit>`，`Hit.method` 是标准反射 `Method`，怎么 Hook 是你的事。
2. **两阶段扫描**：① 纯字节解析 dex 做初筛（不加载类）→ ② 只对候选类反射确认。
3. **FAST vs NORMAL**：
   - `FAST` —— 返回值/参数**字符串精确匹配**，最快，dex 层即可定死。
   - `NORMAL`（默认）—— 弱匹配：`isAssignableFrom`（父类/接口/子类都算）、基本类型与包装类等价。类型条件推迟到反射阶段，候选偏多、要加载类。

---

## 2.5 匹配策略：优先用什么，找不到再用什么

**核心原则：用稳定的东西锚定，别用会变的东西。**

| 稳定（放心用） | 会变（别依赖） |
|---|---|
| 系统类型 `java.lang.String` / `java.util.Map` / `int`… | 混淆的类名 `...X.ME`、方法名 `A05` |
| 父类/接口：`Enum` / `View` / 某接口 | 具体混淆子类名 |
| 参数个数、修饰符、类结构（字段数/方法数） | —— |

**按这个顺序选，命中越少越好：**

1. **首选 · FAST + 精确参数**：参数全用系统类型写全 + 修饰符。dex 层一步命中、不反射、不依赖 classloader。
   ```java
   Ri.m().args(String.class, String.class, Map.class).public_()   // .fast()
   ```
2. **参数不够独特 → 加稳定的返回值/结构条件**（仍 FAST）：
   - `.retSuper(Enum.class)` —— 返回值是某父类的子类（**dex 层沿父类链判定，不写死混淆名**）。
   - `.superCls("父类全名")` / `.hasItf("接口名")` / `.fieldc(n)` / `.methodc(n)` —— 类结构特征。
   ```java
   Ri.m().retSuper(Enum.class).args(String.class, String.class, Map.class).public_()  // .fast()
   ```
3. **必须按“返回值/参数是某父类型”匹配，且 FAST 强匹配不到** → 用 `.normal()`：类型走弱匹配。代价是候选多、要反射，配 `.budget(10000)` 兜时间。
   ```java
   Ri.m().ret(Enum.class).args(String.class, String.class, Map.class).public_()       // .normal()
   ```
   > 注：`.retSuper()` 在 dex 层就收窄，通常比 `.normal() + .ret(父类)` 候选少得多，优先用它。
4. **实在难定位** → 先用 `.probe()` 在本地反复调规则（见 §6），看 dex 层命中几个，调到唯一了再上设备。

**永远别写**：`.name("A05")`、`.ret("com.x.X.ME")` —— 混淆名每个 build 都变。

## 3. OL 入口 API 速查

起链：`OL.with(Context ctx, ClassLoader cl)`

| 分类 | 方法 | 说明 |
|---|---|---|
| **范围** | `.pkg(String)` | **必填**，扫描的类名前缀（也是安全边界）。子包一起扫（`startsWith`） |
| | `.allowAll()` | 允许不设 pkg 全量扫（危险，默认禁止） |
| **模式** | `.fast()` / `.normal()` | 见上。`.deep()/.trace()` 未实现（调用抛异常） |
| | `.mode(Mode)` | 等价显式设模式 |
| **来源** | 默认 | 不设来源就扫宿主 APK（base + split） |
| | `.dex(String... 路径)` | 指定 raw `.dex` 或 apk/jar/zip 文件。设了就**只扫这些** |
| | `.asset(String... 名字)` | 从宿主 assets 读明文 dex/jar（需要 ctx） |
| | `.dexBytes(byte[])` | 直接扫内存字节（加密 assets 解密后用） |
| **进程** | `.anyProc()` | 任意进程都跑。**pkg 不是进程名时必须加**（否则被进程门控跳过） |
| | `.proc(String)` | 只在指定进程名跑 |
| **预算** | `.budget(int ms)` | 时间预算，超时降级 |
| | `.maxClasses(int)` | 类数预算 |
| | `.limit(int)` | 返回/命中上限（默认 20） |
| **缓存** | `.cache(boolean)` | 命中缓存开关（默认开） |
| | `.memOnly()` / `.disk()` | 只内存缓存 / 落磁盘（默认落盘） |
| | `.clear()` | 清掉该 pkg 的缓存 |
| | `.once()` | 同进程内该规则只跑一次 |
| **日志** | `.log()` | 打开库内部调试日志（logcat tag = `OL`） |
| **规则** | `.rule(Rule)` / `.rules(Rule...)` | 预先加规则（配合 `run()`） |

**终止方法（触发执行）：**

| 方法 | 返回 | 说明 |
|---|---|---|
| `.find(rule)` | `List<Hit>` | 加规则并执行 |
| `.run()` | `List<Hit>` | 执行已 `rule()` 加入的规则 |
| `.runWithStats()` | `ScanResult` | 命中 + 扫描统计（`stats`） |
| `.async(cb)` | `void` | 后台线程扫描，完成回调。**仅适合目标稍后才被调用/只需方法位置的场景**；早期热点方法用同步 `find`，否则会漏掉扫描完成前对目标的调用 |
| `.probe(rule)` | `List<ProbeHit>` | **只跑 dex 初筛层**，不反射、不需要设备（见 §6） |

## 4. Ri.m() 规则条件速查

`Ri.m()` 返回方法规则构建器，链式调用；所有条件是**与**关系，不设即“不关心”。

| 分类 | 方法 | 说明 |
|---|---|---|
| **返回值** | `.ret(Class)` / `.ret(String 全名)` | 精确返回值。如 `.ret(void.class)`、`.ret("a.b.C")`、`.ret("[B")` |
| | `.retSuper(Class)` / `.retSuper(String)` | 返回值是**某父类/祖先**的子类（含自身）。dex 层沿父类链判定，**抗混淆改名**，如 `.retSuper(Enum.class)` |
| **参数** | `.args(Class... / String...)` | 精确参数类型列表（隐含数量） |
| | `.argc(n)` / `.min(n)` / `.max(n)` | 只约束参数数量 |
| **修饰符** | `.static_()` `.public_()` `.private_()` `.protected_()` `.native_()` `.synchronized_()` | 无参=必须有；传 `false`=必须没有（三态） |
| **类名** | `.cls(全名)` / `.hasCls(子串)` | 精确 / 包含 |
| **方法名** | `.name(全名)` / `.hasName(子串)` | 精确 / 包含。⚠️ 混淆名每次 build 会变，慎用 |
| **结构**（NORMAL） | `.superCls(全名)` | 父类精确 |
| | `.hasItf(名)` | 实现的接口（任一包含即可） |
| | `.fieldc(n)` / `.methodc(n)` | 字段数 / 方法数 |
| **自定义** | `.filter(Filter)` | 反射阶段对候选自定义收窄（`id()` 要稳定，进缓存 hash） |
| **数量** | `.first()` / `.limit(n)` | 取第一个 / 限量 |

## 5. 返回值

**`Hit`**（反射确认后的最终命中）：

| 字段 | 类型 | 说明 |
|---|---|---|
| `cls` | `String` | 类全名 |
| `name` | `String` | 方法名 |
| `ret` | `String` | 返回值类型名 |
| `args` | `String[]` | 参数类型名 |
| `mod` | `int` | 修饰符位 |
| `method` | `java.lang.reflect.Method` | **拿它去 Hook / 反射调用** |
| `.sig()` | `String` | 可读签名 |

**`ProbeHit`**（`.probe()` 的 dex 层命中，无反射 Method）：`cls / name / ret / params[] / flags`。

## 6. 本地验证规则：`.probe()`（不用设备）

`.probe()` 只跑扫描第一阶段（dex 解析 + dex 层匹配），**不加载类、不反射、不 Hook**，用 `.dex()/.dexBytes()` 时连 `ctx/cl` 都不用传，可以直接在桌面 JVM 单元测试里跑，快速验证“规则写得对不对”。

```java
List<ProbeHit> hits = OL.with(null, null)
        .dex("/path/to/audience_network.dex")   // 或 .dexBytes(byte[])
        .pkg("com.facebook.ads.redexgen.X")
        .anyProc()
        .fast()                                  // FAST：条件在 dex 层定死 → 返回即最终结果
        .probe(Ri.m()
                .args(String.class, String.class, Map.class)  // 稳定：系统类型
                .retSuper(Enum.class)                          // 稳定：返回值是 Enum 子类，不写混淆名
                .public_());
// => [com.facebook.ads.redexgen.X.a5#A05[java.lang.String, java.lang.String, java.util.Map]->...ME flags=0x11]
```

- **FAST**：`probe` 返回的就是最终命中集。
- **NORMAL**：类型条件延后到反射，`probe` 只按 名字/参数数/修饰符/结构 过滤，返回的是**候选**（偏多，用于确认目标没被漏掉）。
- 仓库 `src/test/java/oblocator/DexFindTest.java` 是可在 Android Studio 点绿三角直接跑的例子，`-Doblocator.dex=/path/to/app.apk` 可换任意 dex/apk。

## 7. 扫描动态加载的 dex（assets / 加密）

默认只扫宿主 APK，**扫不到运行时动态加载的 dex**（会 `classes=0`）。三种来源二选一：

```java
// 1) assets 里的明文 dex
OL.with(ctx, cl).asset("audience_network.dex")...

// 2) 已落磁盘的 dex/apk/jar
OL.with(ctx, cl).dex("/data/data/宿主/app_dex/payload.dex")...

// 3) 加密 assets：自己 Hook 拿到解密后的字节
OL.with(ctx, cl).dexBytes(decryptedBytes)...
```

> ⚠️ **`cl` 必须是加载了那个 dex 的 ClassLoader**（例如 FB 自己的 `DexClassLoader`），否则反射阶段 `Class.forName` 全部失败。宿主的 `lpparam.classLoader` 加载不了动态 dex 里的类。

## 8. 完整示例（LSPosed 模块：定位 + 自己 Hook）

以 FB Audience Network 广告 SDK 为例，目标是 `com.facebook.ads.redexgen.X.a5#A05(String,String,Map) -> ME`（`ME extends Enum`）。

```java
public class Entry implements IXposedHookLoadPackage {
    @Override public void handleLoadPackage(XC_LoadPackage.LoadPackageParam lp) {
        if (!"目标宿主包名".equals(lp.packageName)) return;

        // handleLoadPackage 阶段还没 Context，Hook Application.attach 拿：
        XposedHelpers.findAndHookMethod(Application.class, "attach", Context.class,
            new XC_MethodHook() {
                @Override protected void afterHookedMethod(MethodHookParam p) {
                    Context ctx = (Context) p.args[0];
                    ClassLoader dynCl = /* 加载 audience_network.dex 的那个 ClassLoader */;
                    locate(ctx, dynCl);
                }
            });
    }

    private void locate(Context ctx, ClassLoader dynCl) {
        List<Hit> hits = OL.with(ctx, dynCl)
                .asset("audience_network.dex")          // 来源：动态 dex（或 .dex()/.dexBytes()）
                .pkg("com.facebook.ads.redexgen.X")
                .anyProc()                              // pkg 是类前缀不是进程名 → 必加
                .fast()                                 // 条件全稳定 → FAST，dex 层一步命中
                .cache(false)
                .budget(10000)
                .find(Ri.m()
                        .args(String.class, String.class, Map.class)  // 系统类型，稳定锚点
                        .retSuper(Enum.class)           // 返回值是 Enum 子类，不写死混淆的 ME
                        .public_());                    // 排除同签名的 private A00

        for (Hit h : hits) {
            XposedBridge.hookMethod(h.method, new XC_MethodHook() {
                @Override protected void beforeHookedMethod(MethodHookParam param) {
                    // param.args[0..2] 就是那 3 个入参
                }
                @Override protected void afterHookedMethod(MethodHookParam param) {
                    // param.getResult() 是返回的 ME 枚举值
                }
            });
        }
        XposedBridge.log("[OL] located " + hits.size());
    }
}
```

## 9. 常见坑

| 现象 | 原因 | 解法 |
|---|---|---|
| 返回值想写父类（`Enum` 等）FAST 0 命中 | FAST 是字符串精确匹配 | 用 `.retSuper(Enum.class)`（dex 层父类链，优先）；或 `.ret(Enum.class).normal()` |
| NORMAL 候选特别多（上千） | NORMAL 把类型判断推迟到反射，dex 层只按 名/参数数/修饰符 筛 | 加稳定条件收窄：精确 `.args()`、`.retSuper()`、`.superCls()`、`.fieldc()/.methodc()` |
| `scan done: classes=0` | 只扫了宿主 APK，动态 dex 没进来 | 用 `.asset()/.dex()/.dexBytes()` |
| 有候选但 `hits=0`（`load candidate fail`） | `cl` 不是加载该 dex 的 ClassLoader | 换成动态 dex 的 ClassLoader |
| 整个扫描被跳过 | `pkg` 被当进程名做门控 | 加 `.anyProc()` |
| 换个 App 版本就找不到了 | 按混淆方法名/类名匹配，每次 build 会变 | 按**稳定签名/结构**匹配：系统类型参数 + `.retSuper()` + 修饰符，别用 `.name()`/`.ret(混淆名)` |
| 反编译看到的方法名扫不到 | 反编译器会重命名（如 jadx 把 `A05` 显示成别的名） | 以真实 dex 里的名字为准，或按签名匹配 |
